# Hybrid Search RAG
  
A clean, production-style RAG pipeline combining **dense** (BGE-M3), **sparse** (BM25),
and a **cross-encoder reranker** (BGE-reranker-v2-m3), fused with **Reciprocal Rank
Fusion (RRF)**, then generated by a local **Ollama** LLM.

```
┌─────────────┐     ┌─────────┐     ┌────────────────┐
│  Documents  │ ──▶ │ Chunker │ ──▶ │  Embedder      │
│ PDF/MD/TXT  │     │ Recur.  │     │  BGE-M3        │
└─────────────┘     └─────────┘     └───────┬────────┘
                                            │
                             ┌──────────────┴──────────────┐
                             ▼                             ▼
                    ┌─────────────────┐         ┌─────────────────┐
                    │  ChromaDB       │         │  BM25 Index     │
                    │  dense vectors  │         │  sparse weights │
                    └────────┬────────┘         └────────┬────────┘
                             │      Hybrid Retrieve    │
                             └────────────┬────────────┘
                                          ▼ (RRF fusion, top-N)
                               ┌─────────────────────┐
                               │  Reranker           │
                               │  BGE-reranker-v2-m3 │
                               └──────────┬──────────┘
                                          ▼ (top-K)
                               ┌─────────────────────┐
                               │  Ollama LLM         │
                               │  (qwen / gemma …)   │
                               └──────────┬──────────┘
                                          ▼
                                      Answer
```

## Stack

| Layer        | Choice                                              | Why                                              |
| ------------ | --------------------------------------------------- | ------------------------------------------------ |
| Embedding    | `BAAI/bge-m3` (via FlagEmbedding)                   | Multilingual, dense+sparse+colbert in one model  |
| Vector store | ChromaDB (persistent, local)                        | Zero-config, embedded, easy to inspect           |
| Sparse store | `rank_bm25` (pickle on disk)                        | Classical BM25 Okapi, fast & deterministic       |
| Fusion       | Reciprocal Rank Fusion (RRF)                        | Score-free, robust, well-studied                 |
| Reranker     | `BAAI/bge-reranker-v2-m3`                           | Strong multilingual reranker                     |
| LLM          | Ollama local (any chat model, e.g. qwen / gemma)    | Zero API cost, full privacy                      |

## Quick start

```bash
# 1. install (Python 3.12 via uv)
uv sync --extra dev

# 2. drop documents into ./data/raw (pdf / md / txt)
cp -r my-docs/* data/raw/

# 3. start ollama and pull a model
ollama serve &
ollama pull qwen2.5:7b        # or any chat model

# 4. build the index (downloads BGE-M3 + reranker on first run, ~2.5GB)
uv run rag-ingest --data ./data/raw

# 5. ask
uv run rag-query "你的问题"

# 6. (optional) run as an HTTP service
uv run python -m rag.api.server --port 8000
#   POST /query      {"question": "...", "top_k": 5}
#   POST /retrieve   {"question": "..."}   # no LLM
#   GET  /health
```

> The first ingest downloads `BAAI/bge-m3` (~2.27 GB) and
> `BAAI/bge-reranker-v2-m3` (~568 MB) into `./storage/models/`. Subsequent runs are
> fully offline.

## Why Hybrid Search?

Pure dense retrieval misses exact keywords (product SKUs, error codes, names).
Pure BM25 misses semantic paraphrases. Hybrid closes the gap. **RRF** fuses two
ranked lists without needing score normalization — that's why we use it as the
default fusion strategy.

Reranking on top of hybrid further sharpens the top-K. Cost: one extra forward
pass per (query, candidate) pair, but only on ~20–50 candidates, not the whole
corpus.

## Tuning knobs (`config.yaml`)

```yaml
chunking:
  chunk_size: 512         # characters
  chunk_overlap: 64
retrieve:
  top_k_dense: 20         # how many from dense
  top_k_sparse: 20        # how many from BM25
  top_k_final: 5          # after rerank
  rrf_k: 60               # RRF constant
  candidate_pool: 50      # sent to reranker before final slice
rerank:
  enabled: true
  top_n: 30               # rerank this many, then slice top_k_final
generate:
  model: qwen3.5:4b
  temperature: 0.2
  num_ctx: 4096
  timeout: 300            # seconds; qwen3.5 needs longer for first call
```

## Performance (Apple M-series MPS)

- BGE-M3 first load: ~25s (2.27 GB)
- 5 docs → 14 chunks → full embed: ~85s
- Single retrieve + rerank (30 candidates): ~3s
- Ollama qwen3.5:4b generation: ~10–30s (depends on `think`)

## Gotchas (read this before deploying)

1. **transformers pin**: FlagEmbedding 1.2.x is incompatible with `transformers 5.x`
   (it relies on `prepare_for_model` which was removed). `pyproject.toml` caps at
   `<5.0`.
2. **hf_xet**: the new xet downloader can stall behind some proxies. If model
   downloads hang, uninstall it: `uv pip uninstall hf_xet`.
3. **HF cache dir**: by default, set `HF_HOME=./storage/models` (via `.env`) so the
   models live alongside the index — easy to back up and easy to wipe.
4. **qwen3 / qwen3.5 thinking mode**: Ollama returns an empty `content` while
   filling `thinking`. We pass `think=False` in `OllamaGenerator` to avoid that.
5. **Chroma cosine ↔ similarity**: Chroma returns *distance*, not similarity. We
   convert: `similarity = 1 - distance`.

## Extending

- **BGE-M3 lexical weights (sparse from the model)**: enable alongside BM25 for a
  3-way RRF (BGE sparse + BM25 + dense). Set `return_sparse=True` in `BGEM3Embedder`.
- **ColBERT late-interaction**: set `return_colbert_vecs=True` and add a ColBERT
  index (e.g. `pylate`).
- **Evaluation**: wrap retrieval + generation in Ragas (`ragas.metrics.*`).
- **API auth**: add an API-key dependency to `rag/api/server.py` if exposed publicly.
